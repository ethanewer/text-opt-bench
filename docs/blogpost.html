How well does an LLM optimize text?

text-opt-bm gives an LLM agent a weak starting program and a scoring function, then measures how far it can drive the score down by iteratively rewriting the program — across code, prompts, codecs, and data structures.

14 tasks4 families no wall-clock in scoringbit-for-bit replayable gpt-5.5 · 1-hour budget

The benchmark

An agent (a codex loop) edits a program over many iterations to minimize a score. Scores are byte counts, memory (tracemalloc), executed-bytecode counts, or error rates — never wall-clock — so results are hardware-independent and every run replays exactly. Four task families:

  • memory compact data structures scored by serving-time memory — mem_kv / index / graph / intset / str / infer
  • compression lossless codecs scored by output size — compress, compress_heldout
  • algorithms scored by executed bytecode — ops_connect, checkpoint_plan
  • generalization learn from a visible train split; scored on hidden val/test sealed with a secret seed — word_problems, normalize, rule_list, tag_seq, compress_heldout

Experiment 2

Optimization overfits unless the eval data is hidden

The 5 generalization tasks, run two ways and scored on the same validation instances: hidden (the agent sees only an aggregate val score) vs exposed (the val instances are moved into the visible training set, so the agent can read what it is graded on). Both are also judged on an always-hidden test split. Error is shown relative to the seed program (lower = better).

How to read it: train and test are separate plots. In each plot the two lines are the same eval measured under the two conditions — hidden (teal) vs exposed (amber). Overfitting shows up as the amber line diving on train but not following on test.

The exposed gen. gap (exposed test − exposed train) is the overfitting magnitude: how much of the fitted train gain fails to reach held-out test. Largest where data is idiosyncratic and memorizable (rule_list, tag_seq); on the genuinely solvable word_problems the extra data helps instead — the informative exception.
🔒
A visible eval score is untrustworthy: the optimizer will fit it perfectly without generalizing. The defense is structural — hide the eval data and expose only an aggregate score. Every generalization task seals its val/test splits with a secret seed so they cannot be reconstructed.
