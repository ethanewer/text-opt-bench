text-opt-bm: measuring LLM agents as text optimizers

An LLM agent iteratively rewrites a program to minimize a deterministic score. Work in progress; all numbers measured.

The benchmark

Each task hands the agent (a codex loop) a weak starting program and a scoring function; it edits the program over many iterations to drive the score down. Scores are bytes, memory (tracemalloc), executed-bytecode counts, or error rates — no wall-clock, fully deterministic, so runs replay bit-for-bit. 14 tasks in four families:

  • memory compact structures scored by serving-time memory (mem_kv/index/graph/intset/str/infer)
  • compression lossless codecs scored by output size (compress, compress_heldout)
  • algorithms scored by bytecode instructions (ops_connect)
  • generalization learn from visible train, scored on hidden val/test sealed with a secret seed (word_problems, compress_heldout, normalize, rule_list, tag_seq)

Finding 1 — stronger models are better optimizers

Same model gpt-5.5 at two efforts — low (strong) vs none (weak) — 5 runs/task, identical 1-hour budget. Best-so-far score over iterations, normalized per task (1.0 = seed program, 0 = best found), averaged across tasks; seed point dropped:

The strong model descends faster and further; its first accepted program already beats the weak model's final result. Bands = ±1 sd across tasks.
Per-task best-so-far (blue = strong, red = weak), same normalization.
Across all 5×5 run pairs per task, the strong model beats the weak one ~93% of the time (aggregate probability-of-superiority 0.93); 7/8 compared tasks discriminate at ≥0.96. First-attempt quality tracks model strength most cleanly — the weak model partly catches up only by running more, cheaper iterations.

Finding 2 — optimization overfits unless the eval data is hidden

Five generalization tasks, run two ways scored on the same val instances — hidden-val (agent sees only the val score) vs exposed-val (val data moved into the visible training set) — both judged on an always-hidden test split.

Expose the eval data and the agent drives visible val error toward 0, but held-out test doesn't follow (it decouples or worsens). Hidden-val keeps val and test tied.
A visible eval score is untrustworthy — the optimizer fits it perfectly without generalizing. Worst where the data is idiosyncratic/memorizable (rule_list, compress_heldout: test rises); mildest on the genuinely solvable word_problems (more data helps). Control it by hiding the eval data — every generalization task seals val/test with a secret seed.

Finding 3 — noisy scoring flips the strategy

A newer family (in progress) makes the score itself stochastic — sampled from an LM at temperature > 0. From ~1,400 measured probe generations:

A strong model's confident first program (pure reasoning) vs the naive baseline. Under noise, reasoned first attempts land worse than baseline — priors are wrong until you measure.
You can't reason to the answer under noise — you must experiment, average to beat variance (which costs compute: no free lunch), and let measurements correct priors. Where to spend samples becomes the skill.
