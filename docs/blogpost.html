text-opt-bm

A deterministic benchmark that measures how well an LLM agent optimizes text — code, prompts, codecs, data structures — by iteratively rewriting a program to minimize a score.

The benchmark

Each task gives an LLM agent (a codex loop) a weak starting program and a scoring function. The agent edits the program over many iterations to drive the score down; we track the best score reached. Scores are byte counts, memory (tracemalloc), executed-bytecode counts, or error rates — never wall-clock — so every run is hardware-independent and replays bit-for-bit. There are 14 tasks in four families:

  • memory compact structures scored by serving-time memory — mem_kv, mem_index, mem_graph, mem_intset, mem_str, mem_infer
  • compression lossless codecs scored by output size — compress, compress_heldout
  • algorithms scored by executed bytecode — ops_connect, checkpoint_plan
  • generalization learn from a visible train split, scored on hidden val/test sealed with a secret seed — word_problems, normalize, rule_list, tag_seq, compress_heldout

Experiment 2 — optimization overfits unless the eval data is hidden

But the held-out test tells the real story: exposing the eval data drives val toward 0 while test decouples or gets worse. Error normalized to the naive baseline.
A visible eval score is untrustworthy — the optimizer fits it perfectly without generalizing. The effect is largest where the data is idiosyncratic and memorizable (rule_list, compress_heldout: test rises) and smallest on the genuinely solvable word_problems, where more data just helps. The control is structural: hide the eval data and expose only an aggregate score — which is why every generalization task seals its val/test with a secret seed.

All numbers measured on gpt-5.5 low/none, 1-hour deterministic campaigns (5 runs/task). text-opt-bm, work in progress.
