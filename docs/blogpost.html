text-opt-bm: measuring LLM agents as text optimizers

A deterministic benchmark where an LLM agent iteratively rewrites a program to minimize a score. Work in progress. All numbers below are measured.

What the benchmark is

Many useful things are text you can score and improve: a program, a prompt, a compression codec, a data structure. text-opt-bm turns that into a benchmark. Each task gives the agent a weak starting program and a scoring function; the agent (a codex loop running an LLM) edits the program over many iterations, trying to drive the score down. We measure how far it gets.

Two design rules make the results trustworthy: (1) no wall-clock in the score — scores are bytes, tracemalloc memory, executed-bytecode counts, or error rates, so results are hardware-independent and reproducible; and (2) determinism — every task re-scores bit-for-bit (or within a declared tolerance), so a run can be replayed and verified. The agent only ever sees the program and the score signal; it runs in a sandbox with no file/network access.

The tasks (14, four families)

Tasks are built so the winner must be a general algorithm, not a memorized answer table. The families:

  • memory compact data structures scored by serving-time memory (mem_kv/index/graph/intset/str/infer) — you can't fake a measured byte count.
  • compression lossless codecs scored by output size (compress, compress_heldout) — you can't shrink real entropy by memorizing.
  • algorithms scored by executed bytecode instructions (ops_connect) — rewards a genuinely better algorithm.
  • generalization learn from a visible train split, scored on a hidden val/test split (word_problems, compress_heldout, normalize, rule_list, tag_seq). The hidden data is sealed with a secret, uncommitted seed, so it can't be regenerated.

A repeated lesson while building them: a strong LLM one-shots any task with a clean single-algorithm solution, so the interesting tasks have a long tail of idiosyncratic cases that must be discovered incrementally — the score keeps improving over many iterations rather than jumping to the floor at step one.

Finding 1 — stronger models are better text optimizers

We ran the same model, gpt-5.5, at two reasoning efforts — low (strong) vs none (non-reasoning, weak) — on 8 tasks, 5 independent runs each, identical 1-hour budget. Only the model's reasoning changes. For each task we compute the probability of superiority: over all 5×5 run pairs, how often does a strong run beat a weak run? (1.0 = strong wins every pairing; 0.5 = indistinguishable.)

Aggregate probability of superiority = 0.93 (final best) — the stronger model wins ~93% of pairings. 7 of 8 tasks show POS ≥ 0.96; every task discriminates at iteration 1 too.
The benchmark cleanly separates optimizer strength. The gaps are large in magnitude, not just rank: e.g. compress_heldout 1.42e+04 vs 5.95e+04, mem_intset 1.09e+05 vs 1.87e+05, normalize 0.0487 vs 0.114.

One nuance worth presenting: rule_list ties on the final score because the non-reasoning model runs ~4× more (cheaper) iterations per hour and catches up on that slow-converging task — but the strong model still wins per-iteration. So for pure capability ranking, compare at a fixed iteration/token budget, not fixed wall-clock.

Finding 2 — text optimization overfits unless you control for it

This is the failure mode to watch for. We take the generalization tasks and run each in two setups that are scored on the exact same val instances, differing only in whether the optimizer can see those instances:

  • hidden-val (our normal setup): the agent sees the train split and only the val score; the val data is sealed.
  • exposed-val: the val data is moved into the visible training set (we rename train→"debug", val→"train"). Same scoring, but now the agent can fit the data it's graded on.

Both are then evaluated on a third, always-hidden test split — the honest measure of generalization.

Val and test error, normalized to the naive baseline (=1.0). In exposed-val the optimizer drives the visible val score to 0.0 on every task — but the held-out test does not follow it down. In hidden-val, val and test stay tied.

When the optimizer can see the eval set, it fits it perfectly (visible score → 0.0 everywhere) but the held-out test reveals it didn't generalize — a gap of 0.04–0.46 opens between the optimized score and reality. The overfitting is worst where the eval data is idiosyncratic and memorizable (rule_list: exposed test 0.46, worse than the honest hidden-val run's 0.40) and mildest where the task is genuinely solvable, so seeing more labeled data just helps (word_problems: test even improves). The lesson: a visible eval score is untrustworthy — control overfitting by hiding the eval data and exposing only an aggregate score, which is exactly why every generalization task seals its val/test with a secret, uncommitted seed.

Finding 3 — noisy scoring functions change the game

A newer task family (in progress) makes the scoring function itself stochastic: the grade comes from sampling a language model at temperature > 0. The noise is natural, not injected. The agent sees a small-sample estimate; the official grade averages more samples. Designing and probing these (~1,400 measured LM generations) surfaced a sharp finding:

A strong model's confident first program (pure reasoning, before measuring) vs the naive baseline, normalized to baseline=1.0. Higher = worse. On noisy tasks, reasoned first attempts land worse than the naive baseline — the model's priors about what will work are systematically wrong until it measures.
Under noisy scoring, you cannot reason your way to the answer — you must experiment: run, average to beat the variance, and let the measurements correct your priors. And variance reduction is never free — averaging costs compute (measured: Rao–Blackwellization gives no free lunch), so where to spend samples becomes the core skill. These tasks also make one-shot "wins" ambiguous: a lucky first program can harvest headroom by chance, so a fair comparison needs enough samples to separate skill from luck.

Where this is going

The benchmark currently has 14 deterministic tasks that (a) separate strong from weak optimizers, (b) resist one-shotting, and (c) punish overfitting when the eval data leaks. The noisy-scoring family adds a fourth axis — optimizing under measurement uncertainty — and is being hardened before it ships. Everything is reproducible: deterministic scores, sealed held-out data, and full-source audit of every submission.
